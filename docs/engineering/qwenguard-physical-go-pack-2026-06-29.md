# QwenGuard Physical GO Pack 2026-06-29

## Thesis

Issue #106 needs a physical SO-101 session, but the operator should not have to
derive the run order from separate camera, health-check, training, and trial
tools. The physical GO pack consolidates those steps into one non-secret,
phase-driven operator bundle.

This is still preparation. It does not prove SO-101 connectivity, ACT training,
physical manipulation success, latency, reliability, or safety.

## Operator Phases

The generated command script accepts explicit phases:

```bash
bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh fixture
bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh degraded
bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh camera
bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh training-pack
bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh verify
bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh all-safe
```

`all-safe` runs fixture and degraded no-motion traces, generates the LeRobot
ACT training pack, and verifies the generated traces. It does not access the
SO-101 camera and does not move hardware.

The generated script defaults to:

- `RUN_DIR=runs/physical/qwenguard_physical_go`
- `TRAINING_PACK_DIR=runs/physical/qwenguard_so101_training_pack`

Tests or operators may override those with repo-relative paths:

```bash
QWENGUARD_GO_RUN_DIR=runs/physical/session-001/go \
QWENGUARD_TRAINING_PACK_DIR=runs/physical/session-001/training \
  bash runs/physical/qwenguard-physical-go-pack/operator_commands.sh all-safe
```

Absolute paths and `..` path segments are rejected.

The `camera` phase is the first generated phase that touches the SO-101 camera.
The training pack generated by `training-pack` contains the operator-reviewed
LeRobot collection, ACT training, and autonomous evaluation commands, but those
commands are not executed by the physical GO pack itself.

## Motion Interlock Status

The physical GO pack does not yet provide a programmatic interlock from
QwenGuard `ALLOW` to LeRobot actuation. Fixture and degraded traces are
evidence-only: they can prove selector/gate/evaluator behavior, but they do not
authorize motion.

The generated SO-101 training pack now requires manual operator
acknowledgements before motion-capable LeRobot commands:

- emergency stop path reachable and tested;
- low-speed mode selected;
- workspace bounds physically marked and clear;
- leader detached or non-authoritative before autonomous rollout.

Those shell checks are operator acknowledgements, not the final QwenGuard
software interlock. The decision is recorded in
`docs/engineering/qwenguard-motion-interlock-decision-2026-06-29.md`.

## Checked Behavior

The pack generator validates:

- generated shell script syntax;
- repo-contained output path;
- no secret-looking material in generated text or manifest;
- trial template includes `trace_summary_sha`;
- generated paths in the manifest are repo-relative;
- generated evidence template carries the required public-claim gates.

The focused no-hardware test runs the generated `all-safe` phase and verifies
the produced fixture and degraded `decisiontrace.v2` artifacts.

## Non-Claims

- Not SO-101 connectivity proof.
- Not SO-101 camera success.
- Not ACT policy training or physical success.
- Not autonomous physical motion.
- Not a programmatic `ALLOW`-to-LeRobot actuation interlock.
- Not a safety, latency, or reliability claim.
- Not Qwen motor control.
- Not DimOS runtime control.
- Not Alibaba ECS deployment proof.
